- `[manual]` Added "got glutened here" incident reports (`app/server/incidents/`): a login-gated, rate-limited `reportIncident` server function (required `occurredOn` date, optional severity + note), a `listIncidents` read that returns a listing's incidents most-recent-first, and pure recency helpers (`isRecentIncident` / `findRecentIncident`) keyed off a 90-day `RECENT_INCIDENT_WINDOW_DAYS` constant — deliberately separate from the 6-month claim-staleness window (#31). `occurredOn` is validated server-side as a real calendar date (rejecting format-valid-but-impossible dates like `2026-02-31` before they hit Postgres) and cannot be in the future, so a future date can never permanently pin the warning banner. Recency is UTC-calendar-based and `now` is resolved once server-side and threaded into the view, so SSR and client agree (no banner flicker or off-by-one near midnight for the Denver pilot market). On the listing detail page (`app/routes/listings.$id.tsx`) a recent incident raises a prominent warning region near the top (colour + warning-triangle icon + text label) regardless of older confirmations — fresh harm is never buried (ADR-007) — and the "Incident reports" slot lists every report with date and optional severity/note plus a submission form for signed-in visitors. The banner/recency logic is reusable for the browse list-card signal that lands with #33. Client/server boundary is kept clean for the production build: pure helpers + Zod schemas live in `app/trust/incident-recency.ts` (no db import), the client-callable server functions live in `app/server/incidents/incidents.fn.ts` (the `*.fn.ts` convention, so the bundler strips their handlers), and the db-touching implementations stay in `app/server/incidents/index.ts` — so no database/neon code ever enters the browser bundle.
