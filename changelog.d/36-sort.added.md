- `[manual]` Added trust/recency/alphabetical sort to the browse list (#36, domain.md → Discovery). A new accessible, labeled `Sort by` `<select>` on `/listings` is URL-driven via `?sort=` (mirroring the `?page=` pattern: linkable, back/forward works, and changing the sort resets to page 1); selection — never colour — conveys state. Three options: **Alphabetical** (name A–Z, the stable default), **Most trusted** (net confirm consensus — confirms minus disputes — on the headline `celiac_safe_vs_gluten_friendly` claim, then most-recently-confirmed, then name), and **Recently confirmed** (most-recently-confirmed first, then net confirms, then name). The trust/recency orderings use the SAME transparent visible evidence (confirm/dispute distribution + `lastConfirmedAt` recency) the at-a-glance trust derives from — a roll-up of evidence, never an opaque score (ADR-007); unattested listings sort last and every sort ends with a stable name tiebreak. Sort is COMBINABLE with text search (#34): the search predicate is now threaded through the browse loader's `WHERE` (applied to both the page and total-count queries so pagination math stays correct) while sort only changes `ORDER BY`. The sort set lives in a single extensible registry (`app/listings/sort.ts`, client-safe pure) shared by the control and the loader, so #37's "near me" `distance` sort is a clean addition (one registry entry + one `ORDER BY` branch), not a rewrite. Unit tests cover the registry/parser, the per-sort `ORDER BY`, and search+sort combination; the browse Playwright smoke test covers the labeled control and its `?sort=` URL wiring.
