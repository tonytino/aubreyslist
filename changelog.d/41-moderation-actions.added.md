- `[manual]` Added moderation actions (#41, ADR-010) — moderators/admins can now act on flagged content from the #40 queue with **dismiss / hide / remove** (and a server-side **restore**), each safe, soft, reversible, and fully audited. A new `moderation_status` enum column (`visible`/`hidden`/`removed`, default `visible`) on `listings` / `claims` / `incidents` records content state (content is never hard-deleted), and a new append-only `moderation_actions` audit table records who acted, what they did, on which exclusive-arc target (mirroring the `flags` `num_nonnulls(...) = 1` CHECK), the prompting flag, an optional note, and when. A server-only actions module (`app/server/moderation/actions.ts`) gates every action with `requireCurrentRole("moderator")` (admins pass; `user` → 403; anon → 401) BEFORE any DB work, Zod-validates the exclusive-arc input, then writes the audit row + the content-status change + the prompting flag's status (`dismiss`→`dismissed`; `hide`/`remove`→`resolved`) in ONE atomic `db.batch(...)`; client-callable `createServerFn` POST wrappers (`actions.fn.ts`) keep `db` out of the browser bundle. ALL public read paths are now visibility-aware so hidden/removed content disappears and aggregates recompute from visible rows only: the browse list (`browse.ts` — non-visible listings excluded; the celiac aggregate, trust-sort subquery, and recent-incident signal all count visible rows only), listing text search (`search.ts` — hidden/removed listings never returned by name/address search), the GF taxonomy filter (`filter.ts` — only visible claims feed consensus), the listing-detail loader (`get-listing.ts` 404s a removed listing; `listing-summary.ts` drops hidden claims; `incidents/index.ts` drops hidden incidents from the list AND the recent-incident banner), the per-claim aggregate RPC (`attestations/index.ts` — `getClaimAggregate` returns the zeroed/empty aggregate for a hidden/removed/missing claim, never leaking its counts, per ADR-007), directly upholding "recent harm is never buried" (a moderated-away incident drops, a real visible one always stays). The #40 queue UI's disabled placeholder is replaced with real moderator+ Dismiss / Hide / Remove controls (icon SHAPE + text label, never colour alone) wired via TanStack Query `useMutation` that invalidate-and-refetch the queue on success. Vitest covers the permission boundary, the atomic-batch contents per action, exclusive-arc validation, every visibility-aware read path (exclusion + recompute), and the UI wiring; the schema change ships as migration `0001_curvy_captain_britain.sql`.
