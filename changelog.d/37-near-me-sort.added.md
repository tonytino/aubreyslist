- `[manual]` Added a "Near me" distance sort to the browse list (#37, domain.md → Discovery). Picking the new **Near me** option in the `Sort by` `<select>` on `/listings` requests browser geolocation ONLY then (no surprise prompt on load); on grant the list sorts by great-circle (haversine) distance from your location to each listing's stored lat/lng, closest first, with the coords carried in the URL (`?sort=distance&lat=&lng=`) so the view stays linkable/shareable and survives paging. GRACEFUL FALLBACK: if geolocation is denied, unavailable, errors, or times out, the view reverts to the stable alphabetical default and shows an accessible `role="alert"` message (text + neutral chip, never colour alone) explaining the fallback — it never crashes or hangs; the server-side ORDER BY likewise falls back to alphabetical when `sort=distance` arrives without a complete coord pair (denied/SSR), so it can't error. The haversine runs in SQL on the server (ordering by the monotonic haversine term against `listings.lat/lng`); a shared, client-safe pure helper (`app/listings/distance.ts`: `coordsSchema` + `haversineKm`) is the single explainable definition of a valid coordinate and the distance, reused by the loader's validated optional `userLat`/`userLng` input and by the unit tests. The distance sort COMBINES with text search (#34) and the taxonomy filter (#35) — sort only changes `ORDER BY`, so the shared `WHERE`, total count, and pagination stay correct. The geolocation flow lives in a small client-only hook (`app/listings/use-geolocation.ts`) that one-shot-requests location, validates the reading, and resolves grant/deny/unavailable outcomes without throwing. The sort set is still the single extensible registry (`app/listings/sort.ts`) shared by the control and loader, so this was a clean addition (one registry entry + one `ORDER BY` branch). Tests: unit coverage for the coordinate schema + haversine (known distances, symmetry, nearer-ranks-first), the per-sort `ORDER BY` (distance asc + name tiebreak, plus the no-coords/half-pair alphabetical fallback) composed with search + filters, the geolocation hook's grant/deny/unavailable/out-of-range paths, and a Playwright E2E using geolocation mocking (granted coords via context permissions, and a forced deny) that waits for the shared `waitForBrowseReady` hydration guard before interacting so it doesn't race.
