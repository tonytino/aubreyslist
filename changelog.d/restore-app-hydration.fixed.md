- `[propagate]` Fixed the app never hydrating in production builds and dev: the served HTML carried only the SSR markup and CSS, with no `<script type="module">` client entry, so `app/client.tsx`'s `hydrateRoot` never ran and the whole app shipped as a no-JS site (voting, the add-listing form, the taxonomy filter, sort, and SPA `<Link>` navigation were all inert). Root cause: `app/ssr.tsx` did not pass `getRouterManifest` to `createStartHandler`, so the router manifest serialized as `{"$undefined":0}` and `<Scripts/>` emitted no client-entry asset. Wired `getRouterManifest` (from `@tanstack/react-start/router-manifest`) into `createStartHandler`. This is a template-level defect — the construct scaffold's `ssr.tsx` shipped without it — so it should be ported back to `tonytino/construct`.
- `[propagate]` Added a CI build-smoke guard that asserts the served HTML references a `<script type="module">` client entry and that the `/_build/assets/client-*.js` bundle resolves, so a missing client entry (the un-hydrated no-JS regression) can never slip through again — the previous smoke check only verified the stylesheet, which is why this went unnoticed.
- `[propagate]` Added an E2E hydration regression test (`tests/e2e/hydration.spec.ts`) asserting the served HTML includes the client module entry and that a `<Link>` performs client-side (SPA) navigation rather than a full reload.
