- `[manual]` Hardened the add-listing input validator (`createListingInputSchema` in `app/listings/create-input.ts`) so a manual `name`/`address` is `.trim()`-ed BEFORE the `.min(1)` length check (`z.string().trim().min(1)`). Previously `min(1)` ran on the RAW value, so a whitespace-only name/address passed validation and was then trimmed to `""` server-side (`resolveListing`) before insert — bypassing the NOT-NULL-but-empty-permitting columns on a direct server-fn call (the client form already guarded the UI path). The schema now rejects whitespace-only input and emits an already-trimmed value (the subsequent `resolveListing` `.trim()` is now redundant but harmless). Added unit tests for the reject + trim behavior (closes #158).
