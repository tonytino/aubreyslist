- `[manual]` Added the moderation queue (#40, ADR-010) — a moderator/admin-only triage surface inside the admin panel (#38) listing every OPEN flag with context: the target (listing / claim / incident, shown with an icon SHAPE + text label, never colour alone), a human snippet (listing name, claim attribute, or incident note), the reporter (name + email), the free-text reason, and the date filed (newest first, with a link to the target listing). A new server-only query module (`app/server/moderation/queue.ts`) resolves each flag's exclusive-arc target via LEFT JOINs and the reporter via an INNER JOIN in one query; access is enforced SERVER-SIDE by running the real `requireCurrentRole("moderator")` guard (admins out-rank and pass; plain users → forbidden; anonymous → sign-in) and mapping its 401/403 to the same typed `access` discriminator the admin shell uses. The client-callable `createServerFn` wrapper (`queue.fn.ts`) lazily imports that module so `db` stays out of the browser bundle, and the queue UI reads it via TanStack Query (prefetched in the `/admin` loader for no loading flash). Moderation actions (hide / remove / dismiss) are intentionally NOT built here (#41) — each row carries a clearly-labelled, disabled "actions coming soon" placeholder. Vitest covers the access boundary (anonymous → anonymous, user → forbidden, moderator/admin → granted) and the queue shape (listing/claim/incident targets, note-snippet fallback, empty list) with the DB + current-user accessor mocked per `docs/agents/testing.md`.
